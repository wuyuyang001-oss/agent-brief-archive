English overview

ToolPrivacyBench evaluates whether tool-using LLM agents route private information only to tools and downstream sinks that are authorized for the current task purpose. It contains 2,150 executable cases: 1,150 synthetic privacy-sensitive business workflows and 1,000 cases adapted from multi-tool/function-calling benchmarks. Each case is paired with a policy knowledge base, and the evaluator audits recorded tool arguments plus mock-backend logs after execution.

The benchmark’s core claim is that API correctness and workflow completion are insufficient for agent privacy. Across nine evaluated agents, successful task execution can still involve unnecessary disclosure of private atoms through intermediate tools, tickets, notes, notifications, summaries, or handoffs. ToolPrivacyBench therefore formalizes a need-to-know boundary at the field-tool and sink level, with metrics such as FOR, SWLR, ToolFOR, FreeTextFOR, MT-POI, and SMTC.

Its limitation is that it is a controlled audit setting rather than a production incident measurement. The workflows use fabricated or benchmark-adapted values, mock backends, and policy annotations derived from stated tool purposes, so it is best used for comparing mitigations and agent designs rather than estimating real-world breach rates.

对应中文翻译

ToolPrivacyBench 评估 tool-using LLM agents 是否只把私密信息传给当前任务目的下被授权的工具和下游信息接收方。它包含 2,150 个可执行案例：1,150 个合成的隐私敏感业务工作流，以及 1,000 个从多工具/函数调用 benchmark 改造而来的案例。每个案例都配有 policy knowledge base，执行后评测器会审计真实记录下来的工具参数和 mock 后端日志。

论文的核心观点是：API 调用正确、workflow 完成，并不代表 agent 在隐私上合规。九个被评测 agent 中，模型即使完成任务，也可能在中间工具、工单、备注、通知、摘要或交接信息里泄露不必要的 private atoms。因此 ToolPrivacyBench 把“need-to-know”边界形式化到 field-tool 和 sink 级别，并使用 FOR、SWLR、ToolFOR、FreeTextFOR、MT-POI、SMTC 等指标衡量泄露与综合表现。

它的局限是：这是受控审计环境，而不是生产事故率测量。workflow 使用的是虚构或 benchmark 改造数据、mock 后端，以及基于工具目的标注的策略，因此更适合比较防护机制和 agent 设计，而不是直接估计真实世界泄露概率。

方法、创新点、主要结论与局限

核心方法

构造 purpose-bound policy KB，把 private atoms、工具目的、工具 schema、信息 sink 和授权字段关联起来；执行后审计 tool arguments 与 backend logs。

创新点

从“最终回答是否泄露”推进到“多工具轨迹中每个字段是否被送到合适工具/下游”的轨迹级隐私 benchmark。

主要结论

高 TaskSuccess 可以和高 over-disclosure 同时出现；tickets、handoffs、messages、descriptions、work notes 等 free-text sink 是高风险通道。

局限

mock backend 与人工/规则标注政策不能覆盖全部组织角色、留存规则、访问控制、模型更新和人工审核流程。
