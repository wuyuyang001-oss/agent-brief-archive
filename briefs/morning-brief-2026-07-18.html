评测方法 · 新出现同样增加预算，攻击 Agent 与防守 Agent 的收益规律相反

核心结论
作者结论Cybench 的攻击任务会随 test-time compute 增长；BOTS v1 的防守调查更依赖克制的工具使用和正确证据。DeepSeek v4 Flash 在 BOTS 将预算从 $2.10 提到 $4.20 只提升 0.9 pp，95% 配对 bootstrap 区间为 -4.9 到 +6.9 pp。
证据 / 可信度
39 个 Cybench hard challenge、31 个 BOTS v1 计分问题，多模型、三次 epoch、成本拆分和 bootstrap；但运行矩阵是观察性而非随机化，设置并不完全一致。可信度：中。
与既有工作差异
把模型 token、付费工具调用和每单位结果成本放到同一坐标，并把红队与蓝队视为不同 workload，而非一张成功率榜。
产品 / 评测意义
预算上限必须按任务类型设计；防守 Agent 的关键指标应包括证据质量、工具纪律和错误验证，而不是“调用越多越好”。
建议动作

本周最重要的 8 个信号

  1. 行动准确率之外，需要单独测 execute / wait / ask / escalate 的策略校准。
  2. 失败的决定性错误早于可观察失败，轨迹中存在可干预窗口。
  3. 部分 benchmark 是否足够取决于决策阈值与覆盖，不能固定抽 25%。
  4. 领域正确性需要静态、执行、E2E 和语义证据共同定位。
  5. 支付 Skill 配对实验显示平均 +10.31 pp，Skill 可被当作可量化技术资产。
  6. OpenAI 产品信号转向电脑操作、定时和异步执行，评测要覆盖恢复与回滚。
  7. Claude Code 连续版本把预算、后台化、权限和遥测补成 runtime 基础能力。
  8. Skills / harness / web data 项目连续增长，注意力从“模型包装”下沉到工程控制层。

四条趋势线

  • 模型能力：同一 harness 下模型差异仍大，但产品适配和场景稳定性比总均分更能解释边界。
  • Agent 产品：从即时聊天走向 GUI、异步、团队协作；可靠性指标必须跨时间和权限边界。
  • 架构 / 工具：预算、后台任务、Skill、记忆、网页工具和权限解析成为主战场。
  • 评测方法：从 pass/fail 转为 policy、trajectory、decision sufficiency、layered evidence 与 cost-success。

本周论文共同模式与可信度

共同模式是把隐式状态外部化：QANTA 外部化行动阈值，CLI 轨迹外部化错误 onset，部分评测外部化未决比较，PIBench 外部化 rubric 和评测层。可信度整体为中：多篇开放数据、代码或轨迹，但独立复现不足，单作者或单平台偏差明显，统计不确定性不总是充分。

观点增强、削弱与待证

  • 增强“Agent 产品价值来自控制与恢复，而不只是更强模型”被多篇论文和 runtime release 共同支持。
  • 增强“Skill 可以作为产品资产 A/B”被支付 benchmark 的配对结果支持。
  • 削弱“更多任务 / 更多 token / 更多工具调用自然带来更可信结果”被本周证据持续削弱。
  • 待证这些 benchmark 是否能预测真实生产收益，仍缺独立线上研究。

Beyond Success Rate: Cost-Aware Evaluation of Offensive and Defensive Security Agents

作者：Paul Kassianik、Blaine Nelson、Yaron Singer　提交：2026-07-16 17:54 UTC

arXiv · HTML 全文 · PDF · 交互式结果页 · 未发现论文代码仓库链接

选择理由：周末没有新的 arXiv 工作日批次，因此选择最近批次中尚未总结、且最能改变 Agent 产品成本与评测决策的一篇。它不是再给一个安全榜单，而是比较固定预算下的实际 operating points。

此前遇到的问题
安全 Agent benchmark 常在宽松预算下测峰值攻击能力，却忽略每次推理、查询和 enrichment 都有成本；防守场景也被错误地套用“更多 compute 更强”的假设。
研究做法
在共同 Inspect harness 中评估 Cybench hard 攻击任务和 Splunk BOTS v1 防守调查，拆分模型 token 与付费工具成本，比较固定预算、工具调用 cap、拒答和污染控制。
关键结论
攻击任务随预算增长；防守任务不遵循同样规律。Claude Opus 4.8 在 BOTS 以 603 次工具调用得 93.9%，DeepSeek v4 Flash 以 4,450 次调用仅得 73.0%。
与既有工作差异
用 cost-success 曲线、$/1k points、固定预算和 no-tools probe 取代单一峰值成功率。
可信度
中。39 个 Cybench challenge 与 31 个 BOTS 问题、三 epoch、日志日期和 bootstrap 区间公开；但观察性矩阵、模型设置不齐、BOTS v1 公开数据可能污染，未覆盖 v2/v3。
局限
部分 budget cap 是回溯应用；高调用模型会撞 250-message 上限；bootstrap 是描述性稳健检查，不是总体推断；安全任务不能外推一般 Agent。
后续待验证
随机化同配置运行、BOTS v2/v3、真实 SOC 工作流、人工分析师时间、错误告警成本和独立复现。
建议动作
内部 benchmark 同时报告 pass@1、success/$、工具调用分位数、拒答率和无工具污染 probe。
产品机会：为 Agent eval 增加“预算模拟器”：从完整轨迹回放不同 token、工具调用和美元 cap，输出能力曲线、边际收益和推荐部署点。
对抗性审查：表格里 GPT-5.5 在 Cybench 达 94.1%，Claude Opus 4.8 在 BOTS 达 93.9%，但两者不是同一种能力；公开 BOTS 的无工具 probe 仍能得到 50%–74.8%，说明绝对分数可能混入记忆或先验。不能把高分直接写成生产 SOC 就绪。
英文简介与逐段中文对应翻译

English

This paper evaluates security agents through a joint cost-success lens rather than peak success rate alone. Offensive capability is measured on Cybench hard challenges, while defensive investigation is measured on Splunk BOTS v1. Model-token spend and priced external tools are accounted for separately, and completed traces are replayed under fixed budgets and tool-call caps. Offensive CTF performance generally improves with additional test-time compute. Defensive SOC performance does not scale in the same way: stronger results come from disciplined telemetry navigation and selective enrichment rather than raw interaction volume. The study also uses no-tools controls to show that public SOC benchmark scores require contamination checks. The evidence is operationally useful, but the run matrix is observational, model settings vary, and BOTS v1 covers only 31 scored questions from a public dataset.

中文对应翻译

本文不再只看峰值成功率，而是通过“成本—成功”联合视角评估安全 Agent。攻击能力使用 Cybench hard challenges，防守调查使用 Splunk BOTS v1；模型 token 成本与外部付费工具成本被分开核算，完整轨迹还会在固定预算和工具调用上限下回放。攻击型 CTF 表现通常会随 test-time compute 增加；防守型 SOC 表现却不按同样规律扩展，更强结果来自克制的遥测导航和选择性 enrichment，而不是更高交互量。研究还用无工具控制说明，公开 SOC benchmark 的分数必须经过污染检查。这些证据对运营决策有用，但运行矩阵是观察性的、模型设置并不一致，BOTS v1 也只覆盖公开数据中的 31 个计分问题。
